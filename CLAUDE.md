## Read these first ‚Äî fork direction + the SSR feature line

The project has **two roadmaps**, each load-bearing:

1. [`FORK_ROADMAP.md`](./FORK_ROADMAP.md) ‚Äî the master plan. Goals, dependency audit, modernization order, **measurement methodology**, rejected-non-improvements list. **Every change in this fork must follow the change loop in ¬ß Methodology**: capture baseline ‚Üí state numeric hypothesis ‚Üí make change ‚Üí re-measure ‚Üí re-run tests ‚Üí record predicted vs actual.
2. [`PRETEXT_SSR_ROADMAP.md`](./PRETEXT_SSR_ROADMAP.md) ‚Äî the focused single-feature roadmap for the **pretext + SSR + computeLayout** vision. All 6 phases shipped (#015 ‚Üí #020). The fork's headline feature: cascading-grid layout precomputed in pure Node, server emits inline absolute positions, client adopts via `initLayout: false + static: true` ‚Üí measured CLS = 0.00 (down from 0.7421 baseline).

Per-improvement records live in [`improvements/`](./improvements/) (one file per change, using [`improvements/TEMPLATE.md`](./improvements/TEMPLATE.md)). User-facing changes are summarized in [`FORK_RELEASE_NOTES.md`](./FORK_RELEASE_NOTES.md). Both roadmaps' Progress sections track landed work at a glance.

Practical implications:
- **Never make a behavior or size claim without running `scripts/measure.sh` and the visual test suite first.** Predicted improvements that don't show up in real numbers do not land.
- **One roadmap section per commit.** No batching pure deletions with refactors ‚Äî atomic deltas are how the history stays auditable.
- **After every improvement lands, do all four:**
  1. Bump `package.json` version (next `5.0.0-dev.N`).
  2. Update `README.md` ¬ß "Key improvements vs upstream" with a one-liner aimed at *library users* (not maintainers) ‚Äî what they actually get out of this change. Skip changes that have no user-visible effect.
  3. Create an annotated git tag `v5.0.0-dev.N` pointing at the improvement commit, with the improvement title + headline numbers in the tag message.
  4. Mark the row in `FORK_ROADMAP.md` ¬ß Progress (and `PRETEXT_SSR_ROADMAP.md` ¬ß Progress if it's an SSR-feature-line item) as ‚úÖ and link the improvement file. Add a section to `FORK_RELEASE_NOTES.md`.
- **`dist/` is regenerated by the esbuild build script** (`scripts/build.mjs`, landed in #002). Never edit it by hand ‚Äî run `make build` (or `node scripts/build.mjs`) and commit the regenerated artifacts alongside the source change.
- **Don't reintroduce dead browser-compat code.** The fork's whole point is to delete it (vendor prefixes, IE polyfills, matches-selector shim, jQuery shim, etc.). If a deletion breaks something, the right fix is usually a modern API, not restoring the polyfill.
- **The fork's headline feature is `Masonry.computeLayout` + the SSR adoption pattern.** When in doubt about whether a change is on-mission, ask: does it preserve the byte-for-byte agreement test (`test/visual/compute-layout.mjs`) and the measured CLS = 0.00 hydration claim? If yes, it's safe. If it touches the layout math, the existing test gates will catch any regression loudly.

## Project

Masonry (masonry-pretext) ‚Äî a fork of David DeSandro / Metafizzy's cascading grid layout library, modernizing the runtime, fixing dormant upstream pain points, integrating [chenglou/pretext](https://github.com/chenglou/pretext) for reflow-free text measurement, AND shipping the only cascading-grid library on the market with measured zero-flash SSR. Source of truth is the single file `masonry.js` at the repo root. Everything in `dist/` is generated by `scripts/build.mjs`.

## Build, Lint, Test

```sh
npm install                       # ~10 packages, ~13 MB node_modules
npx playwright install chromium   # ~100 MB browser binary, one-time
make build                        # esbuild ‚Üí dist/masonry.{pkgd.js,pkgd.min.js,cjs,mjs}
make test                         # full gate: 10 visual + ssr + module-smoke + compute-layout + no-jquery
make bench                        # server-layout + hydration benchmarks (slow ~2 min, NOT in test gate)
make measure                      # prints size / LOC / dep metrics
```

The historical Gulp 3 / Bower / QUnit toolchain was deleted in improvement #001 and replaced by an esbuild script (#002) that produces four bundle outputs in parallel: `dist/masonry.pkgd.{js,min.js}` (IIFE for `<script>` consumers), `dist/masonry.cjs` (Node `require()`), `dist/masonry.mjs` (modern bundler `import`). See [`improvements/002-esbuild-build.md`](./improvements/002-esbuild-build.md) and [`improvements/013-esm-cjs-builds.md`](./improvements/013-esm-cjs-builds.md).

### Test gates (`make test`)

`make test` runs five separate gates in sequence. Each catches a different regression class:

1. **`test/visual/run.mjs`** ‚Äî custom Playwright runner (not `playwright test`; the CLI runner hangs in this environment, the chromium API works fine). Loads each fixture in `test/visual/pages/`, asserts hardcoded item positions, screenshot-diffs against checked-in baselines in `test/visual/__screenshots__/`. Currently 10 fixtures: `basic`, `gutter`, `horizontal-order`, `fit-width`, `pretext` (#009), `fonts-ready` (#010), `resize-observer` (#012), `static-mode` (#015), `percent-cols` (#014), `init-layout-false` (#018).
2. **`test/visual/ssr-smoke.mjs`** ‚Äî loads `dist/masonry.pkgd.min.js` in a Node `vm` context with empty globals, asserts no crash. Verifies SSR-safety patches from #005.
3. **`test/visual/module-smoke.mjs`** ‚Äî loads `dist/masonry.cjs` via `require()` and `dist/masonry.mjs` via dynamic `import()`, asserts both expose a constructor with `prototype.layout`. Verifies the ESM/CJS bundles from #013.
4. **`test/visual/compute-layout.mjs`** ‚Äî **Node-only** test (no chromium, no JSDOM). Imports `Masonry` from `dist/masonry.mjs` and asserts `Masonry.computeLayout(...)` produces byte-for-byte identical positions to the browser-rendered fixtures for all 9 static cases. **This is the proof that the SSR pipeline math is correct** ‚Äî without it, server‚Üîclient agreement is empirical; with it, it's structural. See [`improvements/017-compute-layout-static-helper.md`](./improvements/017-compute-layout-static-helper.md).
5. **`test/visual/no-jquery.mjs`** ‚Äî greps `dist/masonry.pkgd.{js,min.js}` for any `jquery` / `bridget` strings, asserts zero. Verifies the jQuery removal from #006 doesn't regress.

To add a new visual fixture: drop a self-contained HTML in `test/visual/pages/` that loads `../../../dist/masonry.pkgd.min.js` (no other deps) and sets `window.__READY = true` after init. Then add a case to the `cases` array in `run.mjs`. Use `node test/visual/run.mjs --update --filter=NAME` to capture the baseline. **For features with discriminating semantics (opt-in/opt-out), build a conjugate fixture pair** ‚Äî same setup, opposite expected positions ‚Äî so any regression is caught in both directions. See `static-mode.html` (inverse of `resize-observer.html`) and `init-layout-false.html` for the pattern.

### Benchmarks (`make bench`)

Two benchmarks measure the SSR feature line. **Not in the test gate** because they're slow (~2 minutes) and the chromium one is best-run on a quiet machine.

1. **`test/visual/bench-server-layout.mjs`** ‚Äî pure-Node microbench. Times `Masonry.computeLayout` via `process.hrtime.bigint()` for grids of N=100/500/1000/5000 items. **Current: 0.131 ms median for 5000 items** (38√ó under the 5 ms predicted budget).
2. **`test/visual/bench-hydration.mjs`** ‚Äî Playwright-driven CLS bench. Generates two HTML fixtures at runtime (control = flow-then-relayout, pipeline = inline-positions-from-`computeLayout`), navigates a fresh chromium page to each √ó 30 interleaved runs, captures CLS via `PerformanceObserver({type: 'layout-shift', buffered: true})`. **Current: control CLS = 0.7421, pipeline CLS = 0.0000 (100% reduction)** ‚Äî this is the README headline number.

Both benches share helpers and grid fixture constants from `test/visual/_bench-stats.mjs` (extracted in the simplify pass after #020). The third bench, `test/visual/bench-pretext.mjs` (from #009), measures the client-side pretext fast-path speedup (1.2-1.3√ó faster initial layout) and runs its stats inside `page.evaluate()` so it can't share Node-side helpers.

### Measurement script (`scripts/measure.sh`)

Single source of truth for size, LOC, and dependency-count metrics. Hermetic flags (`gzip -9n`, `brotli -q 11`) so byte numbers are reproducible. Counts tracked + staged + untracked-not-ignored files (i.e., what the next commit would contain). Always run on a clean tree.

```sh
./scripts/measure.sh                            # print to stdout
./scripts/measure.sh --save <label>             # also append a row to metrics/history.tsv
```

## Architecture

Masonry is a thin layout subclass on top of [Outlayer](https://outlayer.metafizzy.co), which provides item lifecycle, events, and the public API (jQuery bridget was removed in #006). `masonry.js` overrides a handful of Outlayer prototype methods PLUS exposes a `Masonry.computeLayout` static method:

### Layout pipeline (`masonry.js`)

- **`_resetLayout`** ‚Äî measures `columnWidth`, `gutter`, computes `cols`, zeros `colYs` (the per-column running Y). Also: detects percentage `columnWidth` (from literal `'20%'` option, inline style, or walked stylesheets) and caches the result behind `options.columnWidth` reference equality. See `detectPercentForOption`. Cache landed in the simplify pass after #020 ‚Äî without it, `_resetLayout` walks `document.styleSheets` on every ResizeObserver-triggered relayout (60+ per second during a window-resize drag).
- **`measureColumns`** ‚Äî delegates to the shared `deriveCols(containerWidth, columnWidth, gutter, percent)` helper to compute `cols` and per-column stride. `deriveCols` handles both the standard branch (with the gutter-overshoot rounding tolerance) and the #014 percent path. Same helper is called by `Masonry.computeLayout` so the math is structurally shared between proto and static.
- **`_getItemLayoutPosition(item)`** ‚Äî DOM adapter (#016 / Phase 1). Resolves `item.size` via `pretextify` (#009) or `item.getSize()` (DOM reflow), then calls the pure-math `placeItem(size, state)` helper.
- **`placeItem(size, state)`** ‚Äî pure-math placement (file-local, not on prototype). Takes pre-measured size + numeric state, mutates `state.colYs` in place, returns `{x, y, col, colSpan}`. **No `this`, no DOM, no option lookups.** Phase 2's `Masonry.computeLayout` calls this same function ‚Äî server and client run identical math.
- **`getTopColPosition` / `getTopColGroup` / `getColGroupY` / `getHorizontalColPosition`** ‚Äî pure helper functions that `placeItem` dispatches to. Also exposed as thin proto wrappers (`proto._getTopColPosition` etc.) for plugin authors who reach into masonry's internals.
- **`_manageStamp`** ‚Äî for stamped (fixed-position) elements: delegates to the shared `applyStamp(colYs, cols, stride, firstX, lastX, stampMaxY)` helper, which is also called by `Masonry.computeLayout` so the stamp colYs-push (with the `#425` off-by-one fix) is structurally shared.
- **`_getContainerSize` / `_getContainerFitWidth`** ‚Äî `_getContainerFitWidth` delegates to the shared `computeFitContainerWidth(cols, colYs, stride, gutter)` helper, again called by `Masonry.computeLayout` for fitWidth output.
- **`needsResizeLayout`** ‚Äî only relayout on resize when the container width actually changed.
- **`Masonry.computeLayout(opts)`** ‚Äî pure-Node static helper. Replicates `_resetLayout` + per-item placement + container-size computation but takes pre-measured sizes as input. Returns `{positions, cols, columnWidth, containerHeight, containerWidth?}`. Fully typed in `masonry.d.ts` (`ComputeLayoutOptions`, `ComputeLayoutResult`). Verified byte-for-byte against the browser layout by `compute-layout.mjs`.

### Constructor extensions in `_create`

`proto._create` is overridden to add three hooks, all gated behind `if (!this.options.static)` so the `static: true` SSR preset (#015) opts out of all of them in one flag:

- **`document.fonts.ready` deferred layout (#010)** ‚Äî when fonts haven't loaded at construction time, schedules a relayout after `document.fonts.ready` resolves. Closes upstream `desandro/masonry#1182`.
- **Per-item `ResizeObserver` (#012)** ‚Äî observes every item element, schedules a relayout via `requestAnimationFrame` coalescing when any item resizes (image loads, font swaps, content edits, parent resizes). Closes upstream `#1147` + 7 image-overlap duplicates. The `_itemize`, `remove`, and `destroy` overrides keep this in sync as items come and go.
- **`static: true` (#015)** ‚Äî sets `transitionDuration: 0` and skips both hooks above. The SSR preset.

### UMD wrapper

UMD wrapper at the top of `masonry.js` supports AMD (`define`), CommonJS (`module.exports`), and browser globals (`window.Masonry`). The AMD branch is slated for deletion in roadmap ¬ß 2.3 ‚Äî until then, keep it intact. Also: `Masonry.compatOptions.fitWidth = 'isFitWidth'` keeps the legacy `isFitWidth` option name working alongside `fitWidth`; use `this._getOption('fitWidth')` rather than reading the option directly.

### Generated bundles

`scripts/build.mjs` produces four `dist/` outputs in parallel via esbuild:

- `dist/masonry.pkgd.js` (IIFE, unminified) ‚Äî for `<script src>` drop-in
- `dist/masonry.pkgd.min.js` (IIFE, minified) ‚Äî same, minified
- `dist/masonry.cjs` (CommonJS) ‚Äî for Node `require()` and webpack 4
- `dist/masonry.mjs` (ESM) ‚Äî for Vite, Rollup, esbuild, webpack 5, Astro, Next.js

Build-time plugins in `scripts/build.mjs` patch the bundled deps to delete dead vendor-prefix code (#004), force SSR safety (#005), strip jQuery branches (#006), delete the get-size box-sizing setup (#007), and delete unused fizzy-ui-utils methods (#008). Never edit the patched files inside `node_modules/` ‚Äî re-run `make build`.

`sandbox/` contains standalone HTML demos that reference `../bower_components/...` paths. These have been broken since the initial fork checkout (Bower deps were never installed in this repo). They're kept as reference for the layouts the library supports ‚Äî porting them to load `dist/masonry.pkgd.min.js` directly is a candidate follow-up but not on the critical path.
